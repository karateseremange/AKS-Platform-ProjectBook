## 1. Objet

Le présent Architecture Decision Record formalise la décision fondatrice relative à l'organisation des référentiels d'AKS Platform.

Il explique pourquoi le projet distingue l'atelier de conception, le dépôt documentaire et le dépôt applicatif, et pourquoi le Project Book constitue l'autorité normative entre la conception et l'implémentation.

---

## 2. Statut de la décision

**Décision acceptée et applicable.**

Cette décision structure durablement la gouvernance, la documentation et le développement d'AKS Platform.

Elle ne doit être remise en cause que si l'organisation retenue ne permet plus d'assurer la traçabilité, la continuité ou la maîtrise du produit.

---

## 3. Contexte

AKS Platform est conçu et développé de manière progressive, à partir d'échanges de conception, de documents de référence et de livrables techniques.

Sans séparation explicite entre ces espaces, plusieurs risques apparaissent :

- une décision peut rester uniquement présente dans l'historique d'une conversation ;
- une proposition abandonnée peut être confondue avec une décision validée ;
- le code peut devenir, par défaut, la seule description du comportement attendu ;
- la compréhension du projet peut dépendre de la mémoire des personnes ayant participé aux échanges ;
- la reprise du projet par un nouveau contributeur peut devenir difficile ;
- la cohérence entre besoin, décision, documentation et implémentation peut être perdue.

Le projet a donc besoin d'une architecture documentaire simple, explicite et durable, adaptée à son mode de fonctionnement réel.

---

## 4. Décision

AKS Platform adopte une architecture fondée sur trois référentiels distincts et complémentaires :

```text
Conversations ChatGPT
Atelier de conception
        │
        ▼
AKS-Platform-ProjectBook
Baseline documentaire officielle
        │
        ▼
AKS-Platform
Référentiel d'implémentation
```

Le flux d'autorité est unidirectionnel :

```text
Conception
    ↓
Documentation
    ↓
Implémentation
```

Les décisions sont élaborées dans l'atelier de conception, formalisées et publiées dans le Project Book, puis implémentées dans le dépôt applicatif.

Aucun de ces référentiels ne se substitue aux deux autres.

---

## 5. Description des référentiels

### 5.1 Atelier de conception

Les conversations ChatGPT constituent l'atelier de conception d'AKS Platform.

Elles permettent notamment :

- d'exprimer et d'analyser les besoins ;
- d'examiner plusieurs solutions ;
- de préparer les décisions ;
- de rédiger et revoir les livrables ;
- d'accompagner leur réalisation.

Elles conservent un rôle exploratoire et collaboratif.

Une conversation, une proposition ou un brouillon ne constitue jamais, à lui seul, une règle officielle du projet.

### 5.2 Project Book

Le dépôt `AKS-Platform-ProjectBook` constitue le référentiel documentaire officiel.

Il contient les décisions, règles, architectures, spécifications et documents de référence nécessaires à la compréhension et au développement du produit.

Seuls les documents publiés dans le Project Book avec le statut `Published` sont normatifs.

### 5.3 Dépôt applicatif

Le dépôt `AKS-Platform` constitue le référentiel d'implémentation.

Il contient le code source, les scripts, les tests et les ressources nécessaires à l'exécution du produit.

Le dépôt applicatif met en œuvre les décisions et exigences publiées dans le Project Book. Il ne définit pas, par sa seule existence, une nouvelle règle normative.

---

## 6. Baseline documentaire

La baseline documentaire correspond à l'état officiel, cohérent et exploitable des documents normatifs publiés dans le Project Book à un instant donné.

Elle constitue la référence commune utilisée pour :

- comprendre les règles applicables ;
- préparer et réaliser les développements ;
- effectuer les revues ;
- vérifier la conformité de l'implémentation ;
- assurer la continuité du projet ;
- établir la traçabilité entre décisions, documents et code.

La baseline documentaire n'est pas l'historique complet des discussions. Elle représente uniquement l'état officiellement retenu et publié.

---

## 7. Motivations

### 7.1 Séparation des responsabilités

Chaque référentiel répond à une responsabilité claire :

- la conversation permet de concevoir ;
- le Project Book permet de formaliser ;
- le dépôt applicatif permet d'implémenter.

Cette séparation réduit les ambiguïtés et évite qu'un même support soit utilisé simultanément comme espace de discussion, documentation officielle et implémentation.

### 7.2 Autorité explicite

Le Project Book fournit une source de vérité identifiable.

Une décision n'est plus considérée comme officielle parce qu'elle a été évoquée, mémorisée ou implémentée, mais parce qu'elle a été formalisée et publiée.

### 7.3 Traçabilité

L'architecture retenue permet de relier :

```text
Besoin
  ↓
Décision
  ↓
Document publié
  ↓
User Story
  ↓
Implémentation
  ↓
Tests
```

Cette chaîne facilite la compréhension des choix et la vérification de leur mise en œuvre.

### 7.4 Continuité du projet

Le projet ne doit pas dépendre exclusivement de l'historique des conversations ou de la mémoire de ses contributeurs.

Le Project Book permet de reprendre le travail à partir d'une base structurée et stable, y compris après une interruption ou lors de l'arrivée d'un nouveau contributeur.

### 7.5 Maintenabilité

Une documentation normative séparée du code facilite l'évolution du produit.

Les règles fonctionnelles et architecturales peuvent être comprises sans devoir déduire leur intention à partir de l'implémentation existante.

### 7.6 Reproductibilité

La distinction entre conception, documentation et code permet de reproduire le raisonnement général du projet sans dépendre de tous les détails historiques ayant conduit à chaque décision.

---

## 8. Options envisagées

### 8.1 Utiliser uniquement les conversations

Cette option a été écartée, car les conversations sont adaptées à l'exploration mais pas à la constitution d'une référence normative stable.

Leur volume, leur caractère chronologique et la présence de propositions successives rendent difficile l'identification de l'état officiel du projet.

### 8.2 Utiliser uniquement le dépôt applicatif

Cette option a été écartée, car le code décrit principalement le comportement implémenté et non l'ensemble des motivations, règles et décisions qui le fondent.

Elle conduirait également à considérer l'implémentation comme source de vérité, y compris lorsqu'elle diverge du besoin validé.

### 8.3 Regrouper documentation et code dans un seul dépôt

Cette option n'est pas retenue comme architecture de référence.

Bien qu'elle puisse convenir à certains projets, elle réduirait ici la séparation entre la baseline documentaire du produit et ses implémentations techniques.

Le maintien de deux dépôts Git distincts clarifie leur responsabilité respective et permet au Project Book de rester exploitable indépendamment du cycle technique du code.

### 8.4 Maintenir des documents sans statut normatif explicite

Cette option a été écartée, car elle ne permettrait pas de distinguer clairement les brouillons, les documents en revue et les références applicables.

Le statut `Published` est donc nécessaire pour identifier les documents officiels.

---

## 9. Conséquences de la décision

### 9.1 Conséquences positives

La décision apporte les bénéfices suivants :

- une source de vérité documentaire clairement identifiée ;
- une séparation nette entre conception, documentation et implémentation ;
- une meilleure traçabilité des décisions ;
- une réduction de la dépendance à l'historique des conversations ;
- une reprise du projet facilitée ;
- une meilleure cohérence entre les User Stories, les documents et le code ;
- une base stable pour les revues, les tests et les évolutions futures.

### 9.2 Contraintes induites

La décision implique également les contraintes suivantes :

- une décision doit être publiée avant de devenir normative ;
- le développement doit référencer les documents applicables ;
- les conversations ne peuvent pas être utilisées comme unique justification d'une implémentation ;
- le code et le Project Book doivent rester synchronisés ;
- une divergence détectée doit être corrigée ;
- la clôture d'une User Story nécessite une documentation à jour.

### 9.3 Coût accepté

Le maintien d'un dépôt documentaire distinct entraîne un effort de rédaction et de synchronisation.

Ce coût est accepté, car il est inférieur au coût futur d'une perte de compréhension, d'une divergence non maîtrisée ou d'une reprise du projet sans référence fiable.

---

## 10. Règles résultantes

La présente décision entraîne les règles suivantes :

1. Les conversations servent à concevoir, mais ne sont jamais normatives.
2. Le Project Book est la source officielle des décisions, règles et spécifications publiées.
3. Le dépôt applicatif implémente le Project Book.
4. Le flux d'autorité va de la conception vers la documentation, puis vers le code.
5. Une User Story référence les documents nécessaires à sa réalisation.
6. Une décision nouvelle ou modifiée est formalisée dans le Project Book.
7. Une divergence entre documentation et implémentation doit être analysée et corrigée.
8. La documentation et le code doivent être cohérents à la clôture d'une User Story.

Les modalités opérationnelles d'application de ces règles sont définies dans les documents de gouvernance correspondants.

---

## 11. Périmètre exclu

Le présent ADR ne définit pas :

- le processus détaillé de publication documentaire ;
- le cycle de vie complet des documents ;
- la stratégie de branches Git ;
- le processus de développement d'une User Story ;
- les critères de qualité ou de revue de code ;
- les mécanismes avancés de gestion de configuration ;
- les Configuration Items ;
- les manifestes ou comparaisons de baselines ;
- les spécifications fonctionnelles ou techniques des modules.

Ces sujets relèvent de documents spécialisés.

---

## 12. Principe de stabilité

Cette décision décrit une architecture fondatrice et non un processus opérationnel susceptible d'évoluer fréquemment.

Elle a vocation à rester stable pendant toute la durée de vie du produit.

Une révision ne serait justifiée que par un changement majeur de l'organisation des référentiels ou de leur autorité respective.

---

## 13. Critères de conformité

L'architecture documentaire est respectée lorsque :

- les conversations restent un atelier de conception ;
- le Project Book contient les documents normatifs publiés ;
- le dépôt applicatif implémente les règles documentées ;
- les User Stories référencent les documents nécessaires ;
- les nouvelles décisions sont formalisées avant d'être considérées comme officielles ;
- les divergences entre documentation et code sont corrigées ;
- la continuité du projet ne dépend pas de la consultation exhaustive des conversations.

---

## 14. Dépendances

`ADR-001` ne dépend d'aucune décision d'architecture antérieure.

Il constitue la décision fondatrice sur laquelle s'appuient les documents de gouvernance documentaire et de développement.

## 16. Conclusion

AKS Platform distingue explicitement l'espace où les décisions sont conçues, le référentiel où elles deviennent normatives et le dépôt où elles sont implémentées.

Le Project Book constitue ainsi le lien d'autorité entre la conception et le code.

Cette architecture garantit une meilleure traçabilité, facilite la continuité du projet et fournit une base durable pour le développement et l'évolution d'AKS Platform.
