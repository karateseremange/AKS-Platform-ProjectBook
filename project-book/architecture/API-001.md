# API-001 — Principes et conventions des API AKS Platform

## 1. Objet

Ce document définit les règles communes applicables aux API exposées ou consommées par AKS Platform.

Il vise à garantir des échanges cohérents, sécurisés, traçables, évolutifs et compatibles avec les différents modules de la plateforme.

## 2. Périmètre

Les règles s’appliquent :

- aux API publiques ;
- aux API d’administration ;
- aux échanges entre AKS Core et les modules métier ;
- aux connecteurs externes, notamment WordPress et Google ;
- aux traitements automatisés utilisant des endpoints applicatifs.

## 3. Principes généraux

Toute API doit respecter les principes suivants :

- contrat explicite et documenté ;
- authentification adaptée au niveau de sensibilité ;
- contrôle d’autorisation systématique ;
- validation stricte des entrées ;
- réponses homogènes ;
- versionnement maîtrisé ;
- idempotence lorsque nécessaire ;
- journalisation sans exposition de données sensibles ;
- compatibilité ascendante autant que possible.

## 4. Organisation des endpoints

Les endpoints doivent être regroupés par domaine fonctionnel.

Exemples :

- `/api/v1/health` ;
- `/api/v1/questionnaires` ;
- `/api/v1/notifications` ;
- `/api/v1/admin/...`.

Les noms utilisent des ressources au pluriel et évitent les verbes dans les chemins lorsque le modèle REST est applicable.

## 5. Versionnement

Les API exposées publiquement ou consommées par un connecteur externe doivent être versionnées.

La version majeure apparaît dans le chemin, par exemple `/api/v1/...`.

Une nouvelle version majeure est requise lorsqu’une modification rompt le contrat existant.

Les changements compatibles peuvent être ajoutés sans changement de version majeure, à condition de ne pas modifier le sens des champs existants.

## 6. Formats d’échange

Le format JSON est utilisé par défaut.

Les dates et heures sont exprimées au format ISO 8601.

Les réponses doivent utiliser un encodage UTF-8.

Les noms de champs doivent rester stables, explicites et cohérents entre les modules.

## 7. Structure des réponses

Une réponse réussie doit contenir les données utiles et, lorsque nécessaire, des métadonnées de pagination ou de contexte.

Une réponse en erreur doit contenir au minimum :

- un code d’erreur stable ;
- un message compréhensible ;
- un identifiant de corrélation ;
- éventuellement des détails de validation non sensibles.

Exemple :

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Certaines données sont invalides.",
    "correlationId": "..."
  }
}
```

## 8. Codes HTTP

Les codes HTTP doivent refléter la nature du résultat.

Principaux cas :

- `200` : succès ;
- `201` : ressource créée ;
- `204` : succès sans contenu ;
- `400` : requête invalide ;
- `401` : authentification absente ou invalide ;
- `403` : accès interdit ;
- `404` : ressource introuvable ;
- `409` : conflit fonctionnel ;
- `422` : données valides syntaxiquement mais rejetées fonctionnellement ;
- `429` : limite de fréquence atteinte ;
- `500` : erreur interne ;
- `503` : service temporairement indisponible.

## 9. Authentification et autorisation

Les mécanismes d’authentification dépendent du contexte :

- session sécurisée pour l’administration ;
- signature de requête pour les connecteurs externes ;
- jeton à durée limitée pour les intégrations autorisées ;
- aucun secret permanent dans le code source ou le navigateur.

L’authentification ne remplace jamais le contrôle d’autorisation.

Chaque endpoint doit vérifier explicitement les droits associés à l’action demandée.

Les exigences détaillées sont définies dans `SECURITY-001`.

## 10. Signature des requêtes externes

Lorsqu’un connecteur externe appelle AKS Platform, la requête peut être protégée par une signature HMAC ou un mécanisme équivalent.

La vérification doit couvrir au minimum :

- le corps de la requête ;
- un horodatage ;
- un identifiant unique ou nonce lorsque nécessaire ;
- une fenêtre temporelle limitée afin d’éviter les rejeux.

## 11. Validation des entrées

Toutes les données reçues doivent être validées côté serveur.

La validation porte notamment sur :

- les types ;
- les formats ;
- les longueurs ;
- les valeurs autorisées ;
- les règles métier ;
- la présence des champs obligatoires.

Les données non prévues par le contrat peuvent être rejetées lorsqu’elles présentent un risque ou une ambiguïté.

## 12. Idempotence

Les opérations susceptibles d’être rejouées doivent être conçues pour éviter les doublons.

Une clé d’idempotence peut être utilisée pour :

- les créations initiées depuis un connecteur ;
- les envois de notifications ;
- les paiements ou actions sensibles futures ;
- les traitements déclenchés par reprise après incident.

Le comportement attendu doit être documenté endpoint par endpoint.

## 13. Pagination, tri et filtrage

Les listes volumineuses doivent être paginées.

Les paramètres de pagination, de tri et de filtrage doivent être cohérents entre les API.

Les limites maximales doivent empêcher l’extraction incontrôlée de volumes excessifs.

## 14. Gestion des erreurs

Les erreurs techniques ne doivent pas exposer :

- les traces internes ;
- les secrets ;
- les requêtes SQL ;
- les chemins système ;
- les données personnelles non nécessaires.

Les erreurs sont corrélées avec les journaux techniques selon `LOG-001` et avec la stratégie définie dans `ERROR-001`.

## 15. Journalisation et audit

Chaque appel significatif doit pouvoir être corrélé à un identifiant unique.

Les journaux peuvent contenir :

- l’endpoint ;
- la méthode ;
- le statut ;
- la durée ;
- l’identité technique ou fonctionnelle ;
- l’identifiant de corrélation.

Les corps complets de requête et de réponse ne doivent pas être journalisés par défaut.

Les actions administratives ou sensibles relèvent également de `AUDIT-001`.

## 16. Limitation de fréquence

Les endpoints publics ou exposés à des intégrations externes doivent pouvoir appliquer une limitation de fréquence.

Cette protection doit limiter les abus sans empêcher les usages légitimes.

Les seuils sont configurables selon `CONFIG-001`.

## 17. Compatibilité et dépréciation

Toute dépréciation doit être annoncée, documentée et accompagnée d’une période de transition raisonnable.

Un endpoint déprécié ne doit pas être supprimé sans solution de remplacement clairement définie.

## 18. Tests

Chaque API doit faire l’objet de tests adaptés :

- cas nominal ;
- erreurs de validation ;
- authentification ;
- autorisation ;
- idempotence ;
- limites de fréquence ;
- compatibilité du contrat.

Les intégrations externes doivent disposer d’un environnement ou d’un mode de test lorsque cela est nécessaire.

## 19. Documentation

Chaque endpoint doit documenter :

- son objectif ;
- sa méthode ;
- son chemin ;
- les prérequis d’authentification ;
- les paramètres ;
- les réponses ;
- les erreurs possibles ;
- les règles d’idempotence ;
- les contraintes de sécurité.

## 20. Responsabilités

AKS Core fournit les mécanismes transversaux communs.

Les modules métier définissent leurs ressources, leurs règles fonctionnelles et leurs contrôles spécifiques.

Les connecteurs respectent les contrats publiés et ne doivent pas dépendre d’un comportement interne non documenté.
