# 1. Objet

Le présent document définit les règles communes applicables aux API exposées, consommées ou utilisées en interne par AKS Platform.

Il vise à garantir des échanges :

- cohérents ;
- sécurisés ;
- traçables ;
- testables ;
- évolutifs ;
- compatibles avec les différents modules et connecteurs de la plateforme.

API-001 complète `ARCH-001` et `CORE-001`. Il définit les conventions de contrat et d’échange sans imposer une technologie d’implémentation particulière.

---

# 2. Position dans le Project Book

API-001 constitue la référence architecturale pour les contrats d’échange applicatifs d’AKS Platform.

Il s’inscrit dans la chaîne documentaire suivante :

```text
ARCH-001
    ↓
CORE-001
    ↓
API-001
    ↓
Contrats propres aux modules et connecteurs
```

Les documents spécialisés complètent API-001 sans pouvoir le contredire, notamment :

- `SECURITY-001` — sécurité ;
- `CONFIG-001` — paramétrage ;
- `LOG-001` — journalisation ;
- `AUDIT-001` — audit ;
- `ERROR-001` — gestion des erreurs ;
- les documents propres aux modules métier et aux connecteurs.

Les règles documentaires générales restent définies par `DOC-001` et `STD-001`.

---

# 3. Périmètre

Les règles du présent document s’appliquent :

- aux API publiques ;
- aux API d’administration ;
- aux API internes exposées par AKS Core ;
- aux échanges entre AKS Core et les modules métier ;
- aux connecteurs externes, notamment WordPress et Google Workspace ;
- aux traitements automatisés utilisant des points d’entrée applicatifs ;
- aux futurs webhooks ou mécanismes équivalents lorsqu’ils seront introduits.

Elles ne décrivent pas les règles métier propres à un module ni le détail technique d’un fournisseur externe.

---

# 4. Catégories d’API

## 4.1 API publiques

Les API publiques sont accessibles depuis un canal exposé à des utilisateurs ou systèmes externes.

Elles exigent un niveau renforcé de validation, de sécurité, de limitation de fréquence et de stabilité contractuelle.

## 4.2 API d’administration

Les API d’administration permettent des opérations réservées aux acteurs autorisés.

Elles doivent appliquer une authentification, une autorisation explicite et une traçabilité adaptée à la sensibilité de l’action.

## 4.3 API internes

Les API internes exposent les services d’AKS Core ou d’un service de plateforme aux modules métier.

Elles constituent des contrats stables. Leur caractère interne ne dispense ni de documentation, ni de validation, ni de gestion de compatibilité.

## 4.4 Connecteurs externes

Les connecteurs adaptent un contrat AKS Platform à un système tiers.

Ils isolent les modules métier des détails propres à WordPress, Google Workspace ou à toute autre API externe.

---

# 5. Principes généraux

Toute API doit respecter les principes suivants :

- contrat explicite et documenté ;
- responsabilité clairement identifiée ;
- authentification adaptée au niveau de sensibilité ;
- contrôle d’autorisation systématique lorsqu’il est requis ;
- validation stricte des entrées côté serveur ;
- réponses homogènes ;
- versionnement maîtrisé ;
- idempotence lorsque nécessaire ;
- journalisation sans exposition de données sensibles ;
- compatibilité ascendante autant que possible ;
- encapsulation des détails techniques ;
- minimisation des données échangées.

Une API ne doit pas devenir un moyen de contourner les responsabilités définies dans `ARCH-001` et `CORE-001`.

---

# 6. Contrat d’API

Chaque contrat doit définir au minimum :

- son objectif ;
- son propriétaire ;
- ses consommateurs autorisés ;
- sa méthode ou son mode d’appel ;
- son point d’entrée ;
- ses données d’entrée ;
- ses données de sortie ;
- ses règles de validation ;
- ses erreurs possibles ;
- ses exigences de sécurité ;
- ses exigences de journalisation ;
- son comportement en cas de répétition ;
- ses garanties de compatibilité ;
- sa version lorsque cela est nécessaire.

Un comportement non documenté ne fait pas partie du contrat et ne doit pas être utilisé comme dépendance durable.

---

# 7. Organisation des endpoints

Les endpoints doivent être regroupés par domaine fonctionnel.

Exemples :

- `/api/v1/health` ;
- `/api/v1/questionnaires` ;
- `/api/v1/notifications` ;
- `/api/v1/admin/...`.

Les noms utilisent des ressources au pluriel et évitent les verbes dans les chemins lorsque le modèle REST est applicable.

Une action qui ne correspond pas naturellement à une ressource peut utiliser une commande explicitement documentée.

---

# 8. Versionnement

Les API publiques, administratives ou consommées par un connecteur externe doivent être versionnées.

La version majeure apparaît dans le chemin lorsque le protocole HTTP est utilisé, par exemple `/api/v1/...`.

Une nouvelle version majeure est requise lorsqu’une modification rompt le contrat existant.

Les changements compatibles peuvent être ajoutés sans changement de version majeure à condition de ne pas :

- supprimer un champ attendu ;
- modifier le sens d’un champ existant ;
- rendre obligatoire un champ auparavant facultatif ;
- modifier un code de résultat de manière incompatible ;
- changer un comportement d’idempotence établi.

Les API internes peuvent utiliser un mécanisme de versionnement différent, mais toute rupture de contrat reste soumise à une analyse d’impact et à une stratégie de migration.

---

# 9. Formats d’échange

Le format JSON est utilisé par défaut pour les échanges structurés sur HTTP.

Les règles suivantes s’appliquent :

- encodage UTF-8 ;
- dates et heures au format ISO 8601 ;
- noms de champs stables, explicites et cohérents ;
- booléens représentés sans ambiguïté ;
- valeurs absentes distinguées des valeurs vides lorsque cette différence est fonctionnelle ;
- unités et formats documentés.

Les formats de fichiers, PDF ou exports structurés relèvent également des contrats documentaires et de génération concernés.

---

# 10. Structure des réponses

Une réponse réussie contient les données utiles et, lorsque nécessaire, des métadonnées de pagination ou de contexte.

Une réponse en erreur contient au minimum :

- un code d’erreur stable ;
- un message compréhensible ;
- un identifiant de corrélation ;
- éventuellement des détails de validation non sensibles.

Exemple :

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Certaines données sont invalides.",
    "correlationId": "..."
  }
}
```

Le détail technique d’une erreur ne doit pas être exposé au consommateur.

---

# 11. Codes HTTP

Les codes HTTP doivent refléter la nature du résultat.

Principaux cas :

- `200` : succès ;
- `201` : ressource créée ;
- `204` : succès sans contenu ;
- `400` : requête invalide ;
- `401` : authentification absente ou invalide ;
- `403` : accès interdit ;
- `404` : ressource introuvable ;
- `409` : conflit ;
- `422` : données valides syntaxiquement mais rejetées fonctionnellement ;
- `429` : limite de fréquence atteinte ;
- `500` : erreur interne ;
- `503` : service temporairement indisponible.

Un module ou connecteur doit interpréter les codes conformément au contrat et ne pas déduire une règle métier d’un détail technique non documenté.

---

# 12. Authentification et autorisation

Les mécanismes d’authentification dépendent du contexte :

- session sécurisée pour l’administration ;
- signature de requête pour les connecteurs externes ;
- jeton à durée limitée pour les intégrations autorisées ;
- mécanisme interne contrôlé pour les services de plateforme.

Aucun secret permanent ne doit être présent dans le code source, le navigateur ou les journaux.

L’authentification ne remplace jamais le contrôle d’autorisation.

Chaque point d’entrée sensible doit vérifier explicitement les droits associés à l’action demandée.

Les exigences détaillées sont définies dans `SECURITY-001`.

---

# 13. Signature des requêtes externes

Lorsqu’un connecteur externe appelle AKS Platform, la requête peut être protégée par une signature HMAC ou un mécanisme équivalent.

La vérification doit couvrir au minimum :

- le corps de la requête ;
- un horodatage ;
- un identifiant unique ou nonce lorsque nécessaire ;
- une fenêtre temporelle limitée afin d’éviter les rejeux.

La comparaison de signature doit utiliser un mécanisme adapté afin de ne pas introduire de vulnérabilité évitable.

Le secret de signature est géré conformément à `SECURITY-001` et `CONFIG-001`.

---

# 14. Validation des entrées

Toutes les données reçues sont validées côté serveur.

La validation porte notamment sur :

- les types ;
- les formats ;
- les longueurs ;
- les valeurs autorisées ;
- la présence des champs obligatoires ;
- la cohérence entre champs ;
- les règles métier relevant du module propriétaire.

Les contrôles techniques communs relèvent d’AKS Core. Les règles métier restent sous la responsabilité du module concerné.

Les données non prévues par le contrat peuvent être rejetées lorsqu’elles présentent un risque ou une ambiguïté.

---

# 15. Idempotence et prévention des doublons

Les opérations susceptibles d’être rejouées doivent être conçues pour éviter les doublons.

Une clé d’idempotence peut être utilisée pour :

- les créations initiées depuis un connecteur ;
- les envois de notifications ;
- les traitements déclenchés par reprise après incident ;
- les futures opérations sensibles nécessitant cette garantie.

Le contrat précise :

- la source de la clé ;
- sa durée de validité ;
- le résultat retourné lors d’un rejeu ;
- les données conservées pour assurer l’idempotence.

---

# 16. Pagination, tri et filtrage

Les listes volumineuses doivent être paginées.

Les paramètres de pagination, de tri et de filtrage doivent être cohérents entre les API.

Le contrat précise :

- la taille par défaut ;
- la taille maximale ;
- les champs triables ;
- les filtres autorisés ;
- le comportement des valeurs invalides.

Les limites maximales empêchent l’extraction incontrôlée de volumes excessifs.

---

# 17. Gestion des erreurs

Les erreurs techniques ne doivent pas exposer :

- les traces internes ;
- les secrets ;
- les chemins système ;
- les détails d’implémentation ;
- les données personnelles non nécessaires.

Les erreurs sont corrélées avec les journaux techniques selon `LOG-001` et avec la stratégie définie dans `ERROR-001`.

Les erreurs fonctionnelles sont définies par le module métier propriétaire et traduites selon la convention commune d’API.

---

# 18. Journalisation et audit

Chaque appel significatif doit pouvoir être corrélé à un identifiant unique.

Les journaux peuvent contenir :

- l’endpoint ou le contrat appelé ;
- la méthode ;
- le statut ;
- la durée ;
- l’identité technique ou fonctionnelle autorisée ;
- l’identifiant de corrélation.

Les corps complets de requête et de réponse ne doivent pas être journalisés par défaut.

Les actions administratives ou sensibles relèvent également de `AUDIT-001`.

---

# 19. Limitation de fréquence

Les endpoints publics ou exposés à des intégrations externes doivent pouvoir appliquer une limitation de fréquence.

Cette protection doit limiter les abus sans empêcher les usages légitimes.

Les seuils sont configurables conformément à `CONFIG-001`.

Le comportement en cas de dépassement doit être explicite et utiliser le code approprié, notamment `429` pour HTTP.

---

# 20. Intégrations externes

Les API de fournisseurs externes sont encapsulées derrière un connecteur ou adaptateur de plateforme.

Le connecteur est responsable :

- de l’authentification technique auprès du fournisseur ;
- de la traduction entre le contrat interne et le contrat externe ;
- de la gestion des quotas et délais d’attente ;
- de la normalisation des erreurs ;
- de la journalisation adaptée ;
- de la limitation des données transmises ;
- de la protection contre une dépendance irréversible au fournisseur.

Un module métier ne doit pas dépendre directement d’un détail propre à une API tierce lorsqu’un contrat AKS Platform existe.

---

# 21. Compatibilité, dépréciation et suppression

Toute dépréciation doit être annoncée, documentée et accompagnée d’une période de transition proportionnée à l’impact.

Un contrat déprécié ne doit pas être supprimé sans :

- solution de remplacement définie ;
- identification des consommateurs ;
- analyse d’impact ;
- plan de migration ;
- mise à jour de la documentation ;
- validation du Product Owner.

Une rupture de compatibilité reste exceptionnelle.

---

# 22. Tests

Chaque API doit faire l’objet de tests adaptés couvrant au minimum les critères applicables suivants :

- cas nominal ;
- erreurs de validation ;
- authentification ;
- autorisation ;
- idempotence ;
- limites de fréquence ;
- codes et structures de réponse ;
- indisponibilité d’une dépendance ;
- compatibilité du contrat ;
- absence d’exposition de données sensibles.

Les intégrations externes doivent pouvoir utiliser un environnement, un mode ou un double de test lorsque cela est nécessaire.

---

# 23. Documentation des contrats

Chaque endpoint ou contrat doit documenter :

- son objectif ;
- son propriétaire ;
- sa méthode ou son mode d’appel ;
- son chemin ou identifiant ;
- les prérequis d’authentification ;
- les autorisations nécessaires ;
- les paramètres et données d’entrée ;
- les réponses ;
- les erreurs possibles ;
- les règles d’idempotence ;
- les contraintes de sécurité ;
- les règles de compatibilité et de versionnement.

La documentation propre à un module respecte `STD-001` et référence API-001 dès qu’elle définit ou consomme un contrat d’API.

---

# 24. Responsabilités

## 24.1 AKS Core

AKS Core fournit :

- les mécanismes transversaux communs ;
- les conventions de réponse et d’erreur ;
- les mécanismes de sécurité partagés ;
- la corrélation et la journalisation communes ;
- les adaptateurs d’intégration validés ;
- les contrats de services de plateforme.

## 24.2 Modules métier

Les modules métier définissent :

- leurs ressources ;
- leurs règles fonctionnelles ;
- leurs validations métier ;
- leurs erreurs fonctionnelles ;
- leurs autorisations spécifiques ;
- les contrats propres à leur domaine.

## 24.3 Connecteurs

Les connecteurs respectent les contrats publiés et ne doivent pas dépendre d’un comportement interne non documenté.

Ils ne contiennent aucune décision métier appartenant à un module.

---

# 25. Critères d’acceptation

Une API ou un contrat est conforme à API-001 lorsque tous les critères applicables suivants sont satisfaits :

- [ ] L’objectif et le propriétaire du contrat sont identifiés.
- [ ] Les entrées, sorties et erreurs sont documentées.
- [ ] Les responsabilités entre Core, module et connecteur sont respectées.
- [ ] Les entrées sont validées côté serveur.
- [ ] L’authentification et l’autorisation sont adaptées au risque.
- [ ] Aucun secret n’est exposé dans le code client ou les journaux.
- [ ] Les réponses et codes sont cohérents.
- [ ] L’idempotence est définie lorsque nécessaire.
- [ ] La journalisation respecte `LOG-001`.
- [ ] La configuration respecte `CONFIG-001`.
- [ ] La sécurité respecte `SECURITY-001`.
- [ ] Les intégrations externes sont encapsulées.
- [ ] La compatibilité et la dépréciation sont maîtrisées.
- [ ] Les tests couvrent les cas nominaux et les erreurs.
- [ ] La documentation du Project Book est à jour.

# 28. Conclusion

API-001 fournit le cadre commun des échanges applicatifs d’AKS Platform.

Il garantit que les API publiques, administratives, internes et les connecteurs reposent sur des contrats explicites, sécurisés, stables et testables.

Son respect permet de préserver :

- l’indépendance des modules ;
- l’encapsulation des services externes ;
- la cohérence des échanges ;
- la sécurité et la traçabilité ;
- la compatibilité des évolutions ;
- la maintenabilité de la plateforme.

Toute nouvelle API ou évolution d’un contrat existant doit rester conforme à `ARCH-001`, `CORE-001` et aux documents spécialisés du Project Book.
