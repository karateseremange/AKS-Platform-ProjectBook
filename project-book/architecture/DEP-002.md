# 1. Objet

Le présent document définit les règles de dépendances applicables aux composants d’AKS Platform.

Il traduit les principes d’architecture de `ARCH-001` en règles concrètes de conception, d’implémentation et de test.

Son objectif est de garantir :

- un faible couplage entre les composants ;
- des dépendances explicites ;
- des responsabilités clairement séparées ;
- une évolution indépendante des modules métier ;
- une testabilité suffisante ;
- l’absence de dépendances circulaires ;
- la maîtrise des intégrations externes.

`DEP-002` constitue la référence pour les développements réalisés à partir de la V1.1.

---

# 2. Principes directeurs

## 2.1 Dépendances explicites

Toute dépendance nécessaire au fonctionnement d’un composant doit être identifiable dans son contrat de création.

Une dépendance doit être fournie :

- par une factory ;
- par un constructeur logique ;
- par un paramètre explicite ;
- ou par le conteneur d’AKS Core lorsqu’un usage centralisé est justifié.

Les dépendances cachées dans des variables globales, des propriétés implicites ou des appels directs dispersés sont interdites.

## 2.2 Dépendance vers une abstraction stable

Un composant doit dépendre d’un contrat minimal et stable plutôt que d’une implémentation complète.

Le consommateur ne doit connaître que les opérations dont il a réellement besoin.

## 2.3 Sens unique des dépendances

Les dépendances suivent le sens défini par l’architecture en couches :

```text
Présentation
    ↓
Métier
    ↓
Plateforme / AKS Core
    ↓
Intégration
    ↓
Infrastructure
```

Une couche inférieure ne dépend jamais d’une couche supérieure.

## 2.4 Absence de dépendance circulaire

Toute dépendance circulaire est interdite.

Lorsqu’un cycle apparaît, il doit être supprimé par l’une des actions suivantes :

- extraction d’un contrat commun ;
- déplacement de la responsabilité vers AKS Core ;
- introduction d’un orchestrateur ;
- publication d’un événement ;
- séparation d’un composant trop large.

## 2.5 Minimisation du graphe de dépendances

Chaque composant dépend du plus petit nombre possible de services.

Une dépendance ne doit pas être ajoutée par commodité lorsqu’une donnée déjà préparée peut être transmise directement.

# 3. Catégories de dépendances

## 3.1 Dépendances obligatoires

Une dépendance obligatoire est nécessaire au fonctionnement nominal du composant.

Elle doit être validée lors de la création du composant.

En son absence, la factory doit échouer immédiatement avec une erreur codifiée et compréhensible.

## 3.2 Dépendances optionnelles

Une dépendance optionnelle apporte une capacité supplémentaire sans empêcher le fonctionnement principal.

Elle doit disposer d’un comportement de repli explicite.

L’absence d’une dépendance optionnelle ne doit jamais provoquer une erreur tardive difficile à diagnostiquer.

## 3.3 Dépendances techniques

Les dépendances techniques concernent notamment :

- la configuration ;
- la journalisation ;
- la gestion de version ;
- la sécurité ;
- les notifications ;
- l’audit ;
- le stockage ;
- les intégrations externes.

Elles sont fournies par AKS Core ou par un adaptateur d’intégration documenté.

## 3.4 Dépendances métier

Un module métier peut dépendre :

- de ses propres composants internes ;
- des services publics d’AKS Core ;
- des contrats d’intégration autorisés.

Il ne dépend jamais de l’implémentation interne d’un autre module métier.

---

# 4. Règles par couche

## 4.1 Présentation

La couche de présentation peut dépendre :

- d’un modèle de vue ;
- d’un contrôleur ou orchestrateur ;
- d’un contrat d’accès à une action métier.

Elle ne doit pas dépendre directement :

- d’une source de données ;
- de `PropertiesService` ;
- de `SpreadsheetApp` ;
- de `DriveApp` ;
- de `GmailApp` ou `MailApp` ;
- d’un service externe ;
- d’une implémentation de stockage.

## 4.2 Métier

La couche métier peut dépendre :

- de services transversaux d’AKS Core ;
- de repositories ou ports métier ;
- de contrats d’intégration ;
- de value objects et règles du même module.

Elle ne doit pas dépendre :

- d’une interface utilisateur ;
- d’un format HTML ;
- d’un onglet ou d’une cellule Google Sheets ;
- d’un détail d’API externe ;
- de l’implémentation interne d’un autre module.

## 4.3 Plateforme / AKS Core

AKS Core peut dépendre :

- de composants techniques communs ;
- d’adaptateurs d’infrastructure ;
- de contrats externes encapsulés.

AKS Core ne doit dépendre d’aucun module métier.

## 4.4 Intégration

La couche d’intégration dépend :

- d’un contrat défini par la plateforme ou un module métier ;
- du SDK ou service externe concerné ;
- de la configuration strictement nécessaire.

Elle traduit les formats externes en objets internes stables.

## 4.5 Infrastructure

La couche infrastructure fournit les moyens d’exécution et de stockage.

Elle ne contient aucune règle métier.

---

# 5. Factories

## 5.1 Rôle

Les factories sont le point de création officiel des composants disposant de dépendances.

Elles doivent :

- recevoir les dépendances ;
- vérifier leur présence ;
- vérifier les opérations minimales attendues ;
- appliquer les valeurs par défaut autorisées ;
- retourner une instance prête à l’emploi ;
- échouer tôt en cas de contrat invalide.

## 5.2 Contrat minimal

La validation porte sur les capacités réellement utilisées.

Exemple : un composant nécessitant uniquement `logger.info()` et `logger.error()` ne doit pas exiger l’ensemble de l’implémentation du logger.

## 5.3 Valeurs par défaut

Une factory peut fournir une dépendance par défaut uniquement lorsque :

- le comportement est déterministe ;
- le repli est documenté ;
- la sécurité n’est pas diminuée ;
- la testabilité reste préservée.

## 5.4 Objets retournés

Les objets publics retournés par les factories doivent être immuables lorsque cela est possible.

L’API publique ne doit pas exposer les dépendances internes.

---

# 6. Conteneur AKS Core

## 6.1 Usage autorisé

Le conteneur peut être utilisé pour :

- enregistrer les services transversaux de la plateforme ;
- résoudre les composants au point de composition ;
- centraliser les dépendances techniques communes ;
- remplacer une implémentation par un double de test.

## 6.2 Usage interdit

Le conteneur ne doit pas être utilisé comme localisateur de services dans le code métier.

Les appels à `resolve()` sont réservés :

- au bootstrap ;
- aux factories de composition ;
- aux points d’entrée de l’application ;
- aux adaptateurs explicitement documentés.

---

# 7. Dépendances externes

Toute dépendance vers un service externe doit être encapsulée dans un adaptateur.

L’adaptateur :

- isole le SDK ou l’API externe ;
- traduit les formats externes ;
- convertit les erreurs techniques en erreurs AKS ;
- peut être remplacé par un double de test ;
- ne contient aucune règle métier.

Les modules métier ne doivent pas appeler directement les services Google Workspace, WordPress ou toute autre API externe.

---

# 8. Dépendances entre modules métier

Les modules métier sont indépendants.

Un module ne peut pas :

- importer l’implémentation interne d’un autre module ;
- lire directement ses données techniques ;
- appeler une fonction privée d’un autre module ;
- dépendre de son interface utilisateur.

Une collaboration entre modules passe par :

- un contrat public stable ;
- un service de plateforme ;
- un événement ;
- un objet de transfert documenté ;
- une orchestration située hors des deux modules.

---

# 9. Gestion des erreurs de dépendance

Une erreur de dépendance doit être détectée au plus tôt.

Elle contient :

- un code stable ;
- le composant concerné ;
- le nom logique de la dépendance ;
- une description compréhensible ;
- aucune donnée sensible.

Exemples de catégories :

- dépendance absente ;
- dépendance invalide ;
- service non enregistré ;
- collision de clé ;
- contrat incomplet ;
- dépendance circulaire détectée.

---

# 10. Testabilité

Chaque composant doit pouvoir être testé sans accès réel aux services externes.

Les tests doivent pouvoir injecter :

- un logger factice ;
- une configuration contrôlée ;
- une horloge déterministe ;
- un adaptateur de stockage en mémoire ;
- un adaptateur de notification factice ;
- un service de version déterministe.

Les tests unitaires ne doivent pas dépendre de l’ordre global de chargement lorsque cet ordre n’est pas le sujet du test.

Les tests d’intégration couvrent séparément le bootstrap et la résolution des services.

---

# 11. Matrice des dépendances

| Consommateur | Peut dépendre de | Ne peut pas dépendre de |
|---|---|---|
| Présentation | modèle de vue, contrôleur | stockage, API externe, configuration technique |
| Module métier | AKS Core public, ports du module | interface, implémentation d’un autre module |
| AKS Core | services techniques, adaptateurs | module métier |
| Administration | services publics AKS Core | détails internes des modules |
| Adaptateur | contrat interne, service externe | règle métier |
| Infrastructure | moyens techniques | logique métier |

# 12. Application aux composants existants

## 12.1 Configuration

Les consommateurs utilisent l’API publique de `AKS.Config`.

L’accès direct aux propriétés d’environnement reste limité au composant de configuration.

## 12.2 Journalisation

Les composants dépendent d’un contrat minimal du logger.

Ils ne dépendent pas du format de stockage des journaux.

## 12.3 Version

Les consommateurs utilisent `AKS.Version` comme source unique.

Le contrat public utilise `releaseName` et non `codename`.

## 12.4 Administration

Le tableau de bord dépend de services préparés et injectés.

Le modèle du tableau de bord ne doit pas résoudre lui-même des services dans le conteneur.

## 12.5 Container

Le Container est utilisé au point de composition et non au cœur des composants.

---

# 13. Contrôle architectural

Toute nouvelle dépendance doit être examinée lors de la revue de code.

La revue vérifie :

- le sens de la dépendance ;
- la nécessité réelle ;
- le contrat minimal ;
- l’absence de cycle ;
- la possibilité de remplacement en test ;
- l’encapsulation des détails externes ;
- l’absence de duplication d’un service AKS Core.

Une dépendance non conforme bloque la validation du livrable.

---

# 14. Plan d’application V1.1

L’application de `DEP-002` suit une approche progressive :

1. corriger les écarts factuels identifiés par `AUDIT-002` ;
2. aligner le tableau de bord sur le contrat `releaseName` ;
3. consolider les validations des factories ;
4. limiter les résolutions du Container aux points de composition ;
5. renforcer les tests d’immuabilité et de dépendances invalides ;
6. appliquer ces règles dès la création d’AKS Analytics ;
7. appliquer les mêmes règles à AKS Calendar.

Aucune refonte massive n’est requise si le code existant respecte déjà le contrat attendu.

---

# 15. Critères d’acceptation

`DEP-002` est considéré comme respecté lorsque :

- les dépendances sont explicites ;
- les factories valident les dépendances obligatoires ;
- le code métier ne résout pas directement les services dans le Container ;
- AKS Core ne dépend d’aucun module métier ;
- les modules métier ne dépendent pas de l’implémentation d’un autre module ;
- les intégrations externes sont encapsulées ;
- les contrats sont minimaux et stables ;
- les composants peuvent être testés avec des doubles ;
- aucune dépendance circulaire n’est présente ;
- les erreurs de composition sont codifiées ;
- les objets publics n’exposent pas leurs dépendances internes ;
- la matrice de dépendances est respectée.

# 17. Conclusion

`DEP-002` formalise les règles permettant de faire évoluer AKS Platform sans transformer AKS Core en monolithe ni créer de couplages implicites entre les futurs modules.

Il établit une discipline simple : dépendances explicites, contrats minimaux, composition centralisée, intégrations encapsulées et tests isolés.
