# 1. Objet

Le présent document définit les règles de dépendances applicables aux composants d’AKS Platform.

Il traduit les principes d’architecture de `ARCH-001` en règles concrètes de conception, d’implémentation et de test.

Son objectif est de garantir :

- un faible couplage entre les composants ;
- des dépendances explicites ;
- des responsabilités clairement séparées ;
- une évolution indépendante des modules métier ;
- une testabilité suffisante ;
- l’absence de dépendances circulaires ;
- la maîtrise des intégrations externes.

`DEP-002` constitue la référence pour les développements réalisés à partir de la V1.1.

---

# 2. Principes directeurs

## 2.1 Dépendances explicites

Toute dépendance nécessaire au fonctionnement d’un composant doit être identifiable dans son contrat de création.

Une dépendance doit être fournie :

- par une factory ;
- par un constructeur logique ;
- par un paramètre explicite ;
- ou par le conteneur d’AKS Core lorsqu’un usage centralisé est justifié.

Les dépendances cachées dans des variables globales, des propriétés implicites ou des appels directs dispersés sont interdites.

## 2.2 Dépendance vers une abstraction stable

Un composant doit dépendre d’un contrat minimal et stable plutôt que d’une implémentation complète.

Le consommateur ne doit connaître que les opérations dont il a réellement besoin.

## 2.3 Sens unique des dépendances

Les dépendances suivent le sens défini par l’architecture en couches :

```text
Présentation
    ↓
Métier
    ↓
Plateforme / AKS Core
    ↓
Intégration
    ↓
Infrastructure
```

Une couche inférieure ne dépend jamais d’une couche supérieure.

## 2.4 Absence de dépendance circulaire

Toute dépendance circulaire est interdite.

Lorsqu’un cycle apparaît, il doit être supprimé par l’une des actions suivantes :

- extraction d’un contrat commun ;
- déplacement de la responsabilité vers AKS Core ;
- introduction d’un orchestrateur ;
- publication d’un événement ;
- séparation d’un composant trop large.

## 2.5 Minimisation du graphe de dépendances

Chaque composant dépend du plus petit nombre possible de services.

Une dépendance ne doit pas être ajoutée par commodité lorsqu’une donnée déjà préparée peut être transmise directement.

# 3. Catégories de dépendances

## 3.1 Dépendances obligatoires

Une dépendance obligatoire est nécessaire au fonctionnement nominal du composant.

Elle doit être validée lors de la création du composant.

En son absence, la factory doit échouer immédiatement avec une erreur codifiée et compréhensible.

## 3.2 Dépendances optionnelles

Une dépendance optionnelle apporte une capacité supplémentaire sans empêcher le fonctionnement principal.

Elle doit disposer d’un comportement de repli explicite.

L’absence d’une dépendance optionnelle ne doit jamais provoquer une erreur tardive difficile à diagnostiquer.

## 3.3 Dépendances techniques

Les dépendances techniques concernent notamment :

- la configuration ;
- la journalisation ;
- la gestion de version ;
- la sécurité ;
- les notifications ;
- l’audit ;
- le stockage ;
- les intégrations externes.

Elles sont fournies par AKS Core ou par un adaptateur d’intégration documenté.

## 3.4 Dépendances métier

Un module métier peut dépendre :

- de ses propres composants internes ;
- des services publics d’AKS Core ;
- des contrats d’intégration autorisés.

Il ne dépend jamais de l’implémentation interne d’un autre module métier.

---

# 4. Règles par couche

## 4.1 Présentation

La couche de présentation peut dépendre :

- d’un modèle de vue ;
- d’un contrôleur ou orchestrateur ;
- d’un contrat d’accès à une action métier.

Elle ne doit pas dépendre directement :

- d’une source de données ;
- de `PropertiesService` ;
- de `SpreadsheetApp` ;
- de `DriveApp` ;
- de `GmailApp` ou `MailApp` ;
- d’un service externe ;
- d’une implémentation de stockage.

## 4.2 Métier

La couche métier peut dépendre :

- de services transversaux d’AKS Core ;
- de repositories ou ports métier ;
- de contrats d’intégration ;
- de value objects et règles du même module.

Elle ne doit pas dépendre :

- d’une interface utilisateur ;
- d’un format HTML ;
- d’un onglet ou d’une cellule Google Sheets ;
- d’un détail d’API externe ;
- de l’implémentation interne d’un autre module.

## 4.3 Plateforme / AKS Core

AKS Core peut dépendre :

- de composants techniques communs ;
- d’adaptateurs d’infrastructure ;
- de contrats externes encapsulés.

AKS Core ne doit dépendre d’aucun module métier.

## 4.4 Intégration

La couche d’intégration dépend :

- d’un contrat défini par la plateforme ou un module métier ;
- du SDK ou service externe concerné ;
- de la configuration strictement nécessaire.

Elle traduit les formats externes en objets internes stables.

## 4.5 Infrastructure

La couche infrastructure fournit les moyens d’exécution et de stockage.

Elle ne contient aucune règle métier.

---

# 5. Factories

## 5.1 Rôle

Les factories sont le point de création officiel des composants disposant de dépendances.

Elles doivent :

- recevoir les dépendances ;
- vérifier leur présence ;
- vérifier les opérations minimales attendues ;
- appliquer les valeurs par défaut autorisées ;
- retourner une instance prête à l’emploi ;
- échouer tôt en cas de contrat invalide.

## 5.2 Contrat minimal

La validation porte sur les capacités réellement utilisées.

Exemple : un composant nécessitant uniquement `logger.info()` et `logger.error()` ne doit pas exiger l’ensemble de l’implémentation du logger.

## 5.3 Valeurs par défaut

Une factory peut fournir une dépendance par défaut uniquement lorsque :

- le comportement est déterministe ;
- le repli est documenté ;
- la sécurité n’est pas diminuée ;
- la testabilité reste préservée.

## 5.4 Objets retournés

Les objets publics retournés par les factories doivent être immuables lorsque cela est possible.

L’API publique ne doit pas exposer les dépendances internes.

# 6. Conteneur AKS Core

## 6.1 Usage autorisé

Le conteneur peut être utilisé pour :

- enregistrer les services transversaux de la plateforme ;
- résoudre les composants au point de composition ;
- centraliser les dépendances techniques communes ;
- remplacer une implémentation par un double de test.

## 6.2 Usage interdit

Le conteneur ne doit pas être utilisé comme localisateur de services dans le code métier.

Les appels à `resolve()` sont réservés :

- au bootstrap ;
- aux factories de composition ;
- aux points d’entrée de l’application ;
- aux adaptateurs explicitement documentés.

Un composant métier ne doit pas rechercher lui-même ses dépendances dans le conteneur.

## 6.3 Enregistrement

Chaque service enregistré doit disposer :

- d’un identifiant stable ;
- d’un propriétaire fonctionnel ou technique ;
- d’une factory documentée ;
- d’un cycle de vie connu ;
- d’un test de résolution.

## 6.4 Cycle de vie

Les cycles de vie autorisés sont :

- singleton de processus lorsque l’état partagé est maîtrisé ;
- instance par résolution ;
- instance fournie explicitement.

Un service mutable ne doit pas être partagé implicitement.

---

# 7. Modules métier

## 7.1 Indépendance

Chaque module métier possède son propre espace de noms et ses propres composants.

Il ne peut utiliser un autre module que par un contrat public explicitement documenté.

## 7.2 Communication entre modules

La communication directe entre modules doit rester exceptionnelle.

Les modes privilégiés sont :

- service commun fourni par AKS Core ;
- événement fonctionnel ;
- orchestrateur de plateforme ;
- contrat de lecture dédié.

## 7.3 Données partagées

Aucun module ne doit lire ou modifier directement le stockage interne d’un autre module.

Les données partagées passent par un contrat stable.

---

# 8. Intégrations externes

## 8.1 Encapsulation

Toute intégration avec Google Workspace, WordPress, Gmail, Calendar ou une API tierce doit être encapsulée derrière un adaptateur.

Le reste de l’application ne dépend pas directement du SDK externe.

## 8.2 Configuration

Les identifiants, URLs, clés et paramètres d’intégration sont fournis par le service de configuration.

Ils ne doivent pas être codés en dur.

## 8.3 Erreurs

Les erreurs externes doivent être traduites en erreurs AKS stables avant de remonter vers les couches métier ou de présentation.

## 8.4 Tests

Les intégrations externes sont remplacées par des doubles de test dans les tests unitaires.

Les tests réels d’intégration sont isolés et identifiés comme tels.

---

# 9. Journalisation, configuration et version

## 9.1 Configuration

Les composants consomment une vue minimale de la configuration.

Ils ne doivent pas connaître le mécanisme de stockage des paramètres.

## 9.2 Journalisation

Le logger est injecté dans les composants qui produisent des événements utiles.

Un composant ne doit pas accéder directement à la console ou à un mécanisme externe lorsqu’un logger AKS est disponible.

## 9.3 Version

Les informations de version sont obtenues via le service public de version.

Les consommateurs utilisent exclusivement les propriétés documentées du contrat public.

Les alias historiques non documentés sont interdits.

Pour la V1.1, la propriété officielle du nom de version est `releaseName`.

---

# 10. Gestion des erreurs

Les erreurs liées aux dépendances doivent :

- être détectées au plus tôt ;
- utiliser un code stable ;
- identifier le composant concerné ;
- ne pas exposer de secret ;
- être journalisées au point approprié ;
- être testables.

Exemples de situations à traiter :

- dépendance obligatoire absente ;
- contrat incomplet ;
- service non enregistré ;
- dépendance circulaire détectée ;
- configuration invalide ;
- intégration indisponible.

---

# 11. Testabilité

## 11.1 Injection de doubles

Toute dépendance externe ou technique doit pouvoir être remplacée par un double de test.

## 11.2 Tests des factories

Chaque factory importante doit être testée pour :

- une création nominale ;
- une dépendance absente ;
- un contrat incomplet ;
- une dépendance optionnelle absente ;
- l’immutabilité de l’API retournée.

## 11.3 Tests du graphe

Les tests d’architecture doivent vérifier progressivement :

- l’absence de dépendance de Core vers un module métier ;
- l’absence d’accès direct à une intégration depuis la présentation ;
- l’absence d’appel au conteneur depuis les composants métier ;
- l’absence de propriété publique obsolète ;
- la cohérence des identifiants de services.

---

# 12. Règles de nommage

Les identifiants de services du conteneur utilisent un nom stable et qualifié.

Format recommandé :

```text
aks.<domaine>.<service>
```

Exemples :

```text
aks.core.config
aks.core.logger
aks.core.version
aks.admin.access
aks.admin.dashboardModel
```

Les factories utilisent un nom explicite contenant `create` ou `factory`.

Les adaptateurs externes indiquent le système concerné dans leur nom.

# 13. Dépendances observées en V1.1

L’audit `AUDIT-002` confirme l’existence des composants transversaux suivants :

- `AKS.Config` ;
- `AKS.Logger` ;
- `AKS.Version` ;
- `AKS.Admin.Access` ;
- `AKS.Admin.DashboardModel` ;
- `AKS.Admin.Dashboard` ;
- `AKS.Core.Container`.

La composition actuelle repose principalement sur des factories et des dépendances injectées, ce qui constitue la base à conserver.

Les points à corriger ou stabiliser sont :

- uniformiser les validations de contrats dans les factories ;
- réserver le conteneur aux points de composition ;
- documenter les identifiants de services ;
- supprimer l’usage de propriétés publiques obsolètes ;
- aligner le tableau de bord sur `releaseName` ;
- ajouter des tests d’architecture et de dépendances.

# 14. Matrice des dépendances autorisées

| Consommateur | Core | Module identique | Autre module | Intégration | Infrastructure |
|--------------|------|------------------|--------------|-------------|----------------|
| Présentation | Oui, via contrôleur | Oui, API publique | Non direct | Non | Non |
| Métier | Oui | Oui | Contrat explicite uniquement | Via port | Non direct |
| AKS Core | Oui | Sans objet | Non | Via adaptateur | Oui |
| Intégration | Oui, minimal | Via contrat | Via contrat | Oui | Oui |
| Infrastructure | Contrats techniques | Non | Non | Oui | Oui |

Toute exception doit être documentée dans une décision d’architecture.

---

# 15. Critères d’acceptation

Le respect de `DEP-002` est validé lorsque :

- toutes les dépendances obligatoires sont explicites ;
- les factories valident leurs contrats ;
- aucun module métier n’est requis par AKS Core ;
- aucun cycle de dépendance n’existe ;
- le conteneur n’est utilisé qu’aux points de composition ;
- les intégrations externes sont encapsulées ;
- les composants peuvent recevoir des doubles de test ;
- les propriétés publiques consommées sont documentées ;
- les erreurs de dépendances sont codifiées ;
- les règles sont couvertes par des tests adaptés.

---

# 16. Dérogations

Une dérogation à `DEP-002` doit :

- être exceptionnelle ;
- être motivée ;
- être limitée dans le temps ;
- identifier le risque ;
- prévoir une action de suppression ;
- être enregistrée dans le registre des décisions.

Aucune dérogation implicite n’est autorisée.

---

# 17. Plan d’application

L’application de `DEP-002` se fait dans l’ordre suivant :

1. corriger les incohérences contractuelles détectées par `AUDIT-002` ;
2. stabiliser les factories existantes ;
3. formaliser les identifiants du conteneur ;
4. ajouter les tests de dépendances ;
5. appliquer les règles aux nouveaux composants ;
6. traiter progressivement les écarts hérités.

La V1.1 ne nécessite pas une réécriture globale de la plateforme.

Les corrections doivent rester incrémentales et sans régression.

# 19. Conclusion

`DEP-002` formalise un modèle de dépendances simple, explicite et adapté à la taille actuelle d’AKS Platform.

Il protège l’indépendance des modules, limite les effets de bord et fournit un cadre concret pour la consolidation technique de la V1.1.

Son application doit rester progressive, pragmatique et compatible avec les fonctionnalités déjà validées.
