# 1. Objet

Le présent document définit la politique de journalisation applicable à AKS Platform.

Il précise les objectifs, les principes, les catégories d'événements, les niveaux, le modèle commun, les règles de sécurité, les modalités de consultation, de conservation et les responsabilités applicables à l'ensemble de la plateforme.

La journalisation constitue un service transversal d'AKS Core. Elle doit fournir une vision fiable de l'activité technique, fonctionnelle, administrative et de sécurité sans devenir une source de données métier parallèle.

---

# 2. Position dans le Project Book

`LOG-001` est le document transversal de référence pour toute production, conservation, consultation ou exploitation de journaux dans AKS Platform.

Il s'inscrit dans le cadre défini par :

- `ADR-001` — Décisions d'architecture ;
- `ARCH-001` — Architecture fonctionnelle ;
- `CORE-001` — Services communs d'AKS Core ;
- `API-001` — Contrats et échanges API ;
- `SECURITY-001` — Principes de sécurité ;
- `CONFIG-001` — Paramétrage centralisé ;
- `ADMIN-001` — Tableau de bord d'administration ;
- `ROADMAP-001` — Trajectoire produit.

Les modules métier doivent appliquer les règles définies dans `LOG-001`. Ils ne doivent pas créer de mécanisme de journalisation isolé, concurrent ou incompatible avec le socle commun.

Les documents spécialisés peuvent préciser les événements propres à leur périmètre, mais ne peuvent pas réduire les exigences du présent document.

---

# 3. Mission

La journalisation doit permettre de :

- suivre l'exécution des traitements ;
- détecter et diagnostiquer les erreurs ;
- retracer les opérations administratives importantes ;
- identifier l'origine d'un incident ;
- contrôler le bon fonctionnement des modules ;
- fournir une information exploitable depuis l'administration ;
- faciliter la maintenance et le support ;
- contribuer à la sécurité et à la conformité de la plateforme ;
- permettre la corrélation d'événements appartenant à une même opération ;
- disposer d'éléments factuels pour analyser une anomalie.

La journalisation ne remplace ni :

- les données métier ;
- les historiques fonctionnels propres aux modules ;
- les sauvegardes ;
- les mécanismes de reprise ;
- les preuves d'audit nécessitant une conservation spécifique.

# 4. Principes de conception

## 4.1 Centralisation

Tous les modules utilisent un mécanisme commun fourni ou piloté par AKS Core.

Cette centralisation garantit :

- une structure homogène ;
- des règles de sécurité communes ;
- une consultation cohérente ;
- une maintenance simplifiée ;
- une évolution maîtrisée ;
- une politique de conservation cohérente ;
- une intégration uniforme des futurs modules.

## 4.2 Proportionnalité

Seules les informations utiles à l'exploitation, à la sécurité, au diagnostic ou à la traçabilité doivent être enregistrées.

La journalisation ne doit pas collecter des données par défaut sans objectif identifié.

Le volume généré doit rester proportionné à l'utilité opérationnelle des événements.

## 4.3 Lisibilité

Chaque événement doit être compréhensible sans nécessiter l'analyse du code source.

Le message doit permettre d'identifier clairement :

- ce qui s'est produit ;
- le composant concerné ;
- le résultat obtenu ;
- l'impact éventuel ;
- l'action éventuellement attendue.

## 4.4 Structure

Les événements doivent utiliser des champs structurés et stables.

Le texte libre complète le modèle commun mais ne doit pas constituer l'unique information exploitable.

## 4.5 Corrélation

Les événements appartenant à une même opération doivent pouvoir être rapprochés grâce à un identifiant de corrélation.

Cet identifiant doit suivre le traitement depuis son déclenchement jusqu'à son résultat final, y compris lorsqu'il traverse plusieurs composants, modules ou intégrations externes.

Lorsqu'une requête entrante fournit un identifiant de corrélation valide, celui-ci peut être propagé. Dans le cas contraire, la plateforme doit en générer un.

## 4.6 Extensibilité

Le mécanisme commun doit pouvoir accueillir de nouveaux modules sans modifier les principes définis dans ce document.

Chaque nouveau module doit déclarer ses catégories, ses types d'événements significatifs et ses éventuelles règles de conservation spécifiques dans le respect du modèle commun.

## 4.7 Indépendance fonctionnelle

Une défaillance du mécanisme de journalisation ne doit pas provoquer la perte ou l'altération silencieuse d'une opération métier.

Le comportement de repli doit être explicite, maîtrisé et proportionné à la criticité de l'événement.

Une impossibilité de journaliser un événement de sécurité ou d'administration critique doit être signalée et traitée comme une anomalie.

## 4.8 Refus de la duplication

Un module ne doit pas recopier durablement les journaux dans sa propre source de données, sauf besoin fonctionnel ou réglementaire explicitement documenté.

# 5. Architecture fonctionnelle

La journalisation repose sur quatre fonctions complémentaires :

1. **Production** : les composants et modules émettent des événements structurés.
2. **Centralisation** : AKS Core applique un modèle, des règles et un point de traitement communs.
3. **Conservation** : les événements sont conservés selon leur nature, leur utilité et la politique applicable.
4. **Consultation** : les utilisateurs autorisés accèdent aux événements depuis les fonctions d'administration.

Le système commun doit distinguer au minimum :

- les événements techniques ;
- les événements de sécurité ;
- les événements d'administration ;
- les événements fonctionnels significatifs ;
- les événements liés aux échanges externes ;
- les événements d'audit lorsque leur valeur probante ou leur durée de conservation diffère des journaux ordinaires.

Cette séparation facilite le filtrage, la recherche, la conservation et le contrôle des accès.

`LOG-001` définit la journalisation commune. Les exigences d'audit détaillées, lorsqu'elles nécessitent des garanties supplémentaires d'intégrité, de conservation ou de consultation, relèvent de `AUDIT-001`.

# 6. Catégories d'événements

## 6.1 Événements techniques

Ils décrivent le fonctionnement interne de la plateforme, notamment :

- démarrage ou arrêt d'un traitement ;
- résultat d'une opération ;
- erreur d'exécution ;
- indisponibilité d'un service ;
- échec d'accès à une ressource ;
- anomalie détectée par un composant ;
- dépassement de délai ;
- reprise automatique d'une opération.

## 6.2 Événements de sécurité

Ils concernent notamment :

- tentative d'accès refusée ;
- échec d'authentification ou d'autorisation ;
- signature ou contrôle d'intégrité invalide ;
- secret ou configuration de sécurité manquante ;
- comportement anormal détecté ;
- opération sensible exécutée ;
- violation d'une règle de sécurité ;
- protection contre le rejeu déclenchée ;
- révocation ou rotation d'un accès technique.

## 6.3 Événements d'administration

Ils retracent notamment :

- modification d'un paramètre ;
- activation ou désactivation d'une fonction ;
- changement d'état d'un module ;
- lancement d'une opération administrative ;
- archivage ou suppression contrôlée ;
- export de données ou de journaux ;
- action nécessitant une traçabilité particulière.

## 6.4 Événements fonctionnels

Ils décrivent les étapes significatives d'un processus métier sans reproduire les données métier détaillées.

Ils peuvent notamment concerner :

- création, ouverture ou clôture d'une campagne ;
- réception d'une demande ;
- décision administrative ;
- production d'un document ;
- envoi d'une notification ;
- changement d'état important ;
- fin d'un traitement métier avec son résultat synthétique.

## 6.5 Événements d'intégration

Ils concernent les échanges avec des systèmes externes, notamment :

- appel d'une API ;
- réponse reçue ;
- délai dépassé ;
- erreur de communication ;
- rejet d'une requête ;
- indisponibilité d'un service tiers ;
- échec d'une signature ou d'un contrôle d'origine ;
- reprise ou abandon d'un échange.

Les contenus complets des échanges externes ne doivent pas être journalisés lorsqu'ils contiennent des données personnelles, des secrets ou des informations sensibles.

---

# 7. Événements obligatoires

Les événements suivants doivent être journalisés lorsqu'ils existent dans un processus :

- début d'un traitement important ;
- fin du traitement et résultat ;
- erreur empêchant l'exécution normale ;
- avertissement nécessitant une surveillance ;
- modification d'un paramètre administrable ;
- opération administrative sensible ;
- refus d'accès ou contrôle de sécurité échoué ;
- échange externe en erreur ;
- génération ou envoi d'un document important ;
- suppression, archivage ou export de données ;
- changement d'état majeur d'un module ou d'une campagne ;
- échec de journalisation d'un événement critique.

Un traitement fréquent et sans enjeu particulier peut être journalisé de manière synthétique afin d'éviter un volume inutile.

La liste des événements obligatoires propres à un module doit être documentée dans le document fonctionnel ou technique du module.

---

# 8. Niveaux de journalisation

Les niveaux applicables sont les suivants :

## 8.1 Débogage

Le niveau **Débogage** fournit des informations détaillées utiles au diagnostic technique.

Il doit être désactivé ou strictement limité en production, sauf activation temporaire, autorisée et tracée.

Il ne doit jamais entraîner l'exposition de secrets ou de données personnelles non nécessaires.

## 8.2 Information

Le niveau **Information** décrit une opération normale ou un événement utile au suivi de l'activité.

Exemples :

- traitement terminé avec succès ;
- campagne ouverte ;
- notification envoyée ;
- paramètre mis à jour.

## 8.3 Avertissement

Le niveau **Avertissement** signale une situation inhabituelle qui ne bloque pas immédiatement le fonctionnement mais nécessite une attention.

Exemples :

- donnée facultative absente ;
- service externe temporairement ralenti ;
- tentative automatiquement reprise ;
- configuration incomplète mais non bloquante.

## 8.4 Erreur

Le niveau **Erreur** signale l'échec d'une opération ou une anomalie affectant un traitement.

Exemples :

- document non généré ;
- notification non envoyée ;
- traitement interrompu ;
- accès à une ressource impossible.

## 8.5 Critique

Le niveau **Critique** signale une situation susceptible d'affecter fortement la disponibilité, l'intégrité, la sécurité ou la continuité de la plateforme.

Exemples :

- service essentiel indisponible ;
- intégrité compromise ;
- configuration de sécurité invalide ;
- échec généralisé d'un traitement essentiel ;
- impossibilité de tracer une opération administrative critique.

Le niveau doit refléter l'impact réel de l'événement. Une utilisation excessive des niveaux élevés réduit leur efficacité opérationnelle.

Le niveau ne doit pas être déterminé uniquement par la nature technique de l'erreur, mais par son impact fonctionnel, opérationnel et de sécurité.

---

# 9. Modèle commun d'un événement

Chaque événement doit pouvoir contenir les informations suivantes :

- date et heure ;
- fuseau horaire ou référence temporelle commune ;
- identifiant unique de l'événement ;
- identifiant de corrélation lorsqu'il existe ;
- catégorie ;
- niveau ;
- origine ou composant émetteur ;
- module concerné ;
- type ou code stable d'événement ;
- version du modèle d'événement lorsque nécessaire ;
- description synthétique ;
- résultat ou état ;
- contexte technique strictement nécessaire ;
- identifiant de l'acteur lorsque sa conservation est justifiée ;
- type d'acteur : utilisateur, administrateur, service ou traitement automatique ;
- référence fonctionnelle non sensible lorsque nécessaire ;
- durée du traitement lorsque pertinente ;
- informations utiles au diagnostic ;
- indicateur d'environnement lorsque plusieurs environnements existent.

Les champs non applicables peuvent rester absents. La structure générale doit toutefois rester compatible entre les modules.

Les messages doivent être stables, exploitables et adaptés à la recherche. Ils ne doivent pas dépendre uniquement d'un texte libre non structuré.

Le type ou code d'événement doit rester stable afin de permettre le filtrage, les statistiques et les contrôles automatisés.

---

# 10. Horodatage et identifiants

Les événements doivent utiliser une référence temporelle cohérente avec le fuseau horaire défini dans `CONFIG-001`.

Lorsque le support de stockage le permet, l'horodatage technique doit être conservé dans un format non ambigu, puis présenté dans le fuseau horaire attendu par l'administration.

Les identifiants d'événement et de corrélation doivent :

- être suffisamment uniques dans le périmètre de la plateforme ;
- ne pas contenir de donnée personnelle ;
- ne pas révéler de secret ou d'information interne sensible ;
- rester stables pendant toute la durée du traitement concerné.

---

# 11. Données interdites ou restreintes

Ne doivent jamais être journalisés en clair :

- mots de passe ;
- clés privées ;
- jetons d'accès ;
- secrets applicatifs ;
- signatures complètes ;
- données d'authentification ;
- contenus de fichiers confidentiels ;
- réponses détaillées au questionnaire santé ;
- données médicales ou de santé non indispensables ;
- corps complet d'une requête contenant des données personnelles ;
- documents générés ou pièces jointes ;
- valeurs complètes de paramètres sensibles ;
- cookies ou en-têtes d'authentification ;
- données permettant de reconstituer un secret.

Les données personnelles ne peuvent être enregistrées que lorsqu'elles sont strictement nécessaires à un objectif de traçabilité défini.

Lorsqu'une référence suffit, le journal doit utiliser un identifiant technique ou fonctionnel plutôt que reproduire le contenu de la donnée.

Toute information sensible nécessaire au diagnostic doit être masquée, tronquée ou remplacée par une valeur non réversible adaptée à l'usage.

Le masquage doit être appliqué avant l'enregistrement de l'événement, et non uniquement lors de son affichage.

---

# 12. Sécurité et intégrité

Le système de journalisation doit :

- appliquer les droits d'accès définis par la plateforme ;
- limiter la consultation aux utilisateurs autorisés ;
- protéger les événements contre les modifications non autorisées ;
- empêcher l'exposition de secrets ;
- éviter les injections de contenu dans les messages ;
- conserver une date et une heure cohérentes ;
- distinguer l'événement d'origine de son affichage ;
- permettre l'identification des opérations sensibles ;
- limiter les possibilités de suppression ou d'altération ;
- tracer les opérations portant sur les journaux eux-mêmes.

Les journaux ne doivent pas être utilisés pour contourner les règles d'accès aux données métier.

La suppression, l'archivage ou l'export des journaux constitue une opération administrative contrôlée et doit elle-même être traçable.

Lorsqu'un événement possède une valeur d'audit particulière, son intégrité et sa conservation doivent respecter les exigences complémentaires définies dans `AUDIT-001`.

---

# 13. Consultation

Le tableau de bord d'administration défini dans `ADMIN-001` doit permettre, selon les droits accordés :

- d'afficher les événements récents ;
- de filtrer par période ;
- de filtrer par niveau ;
- de filtrer par catégorie ;
- de filtrer par module ou composant ;
- de filtrer par type d'événement ;
- de rechercher un identifiant de corrélation ;
- de consulter le détail d'un événement ;
- d'identifier rapidement les erreurs et événements critiques ;
- de distinguer les événements techniques des opérations administratives ;
- de distinguer les événements actifs des événements archivés lorsque cela est applicable.

L'interface doit privilégier la lisibilité et la recherche opérationnelle. Elle ne doit pas exposer une représentation brute susceptible de révéler des informations sensibles.

Les capacités d'export éventuelles doivent respecter les mêmes règles d'autorisation, de confidentialité et de conservation que la consultation en ligne.

Les exports doivent eux-mêmes être journalisés lorsqu'ils concernent des événements sensibles ou un volume significatif.

---

# 14. Conservation et cycle de vie

La durée de conservation dépend :

- de la catégorie de l'événement ;
- de son niveau ;
- de son utilité opérationnelle ;
- des besoins de sécurité ;
- des obligations applicables ;
- du volume produit ;
- de son éventuelle valeur d'audit.

Une politique de conservation doit définir :

- la durée de conservation active ;
- les règles d'archivage ;
- les conditions de suppression ;
- les éventuelles exceptions ;
- les responsabilités de contrôle ;
- les modalités de purge ;
- les règles applicables aux exports.

Les journaux devenus inutiles doivent être supprimés ou archivés de manière contrôlée.

La conservation ne doit pas être illimitée par défaut.

Les paramètres associés à la conservation relèvent de `CONFIG-001` lorsqu'ils sont administrables.

Une exception de conservation doit être justifiée, limitée dans le temps, approuvée et documentée.

La suppression doit éviter de laisser des copies résiduelles non maîtrisées dans des exports ou supports temporaires.

---

# 15. Volumétrie et performance

Le mécanisme de journalisation doit rester compatible avec les limites de l'environnement d'exécution.

Les règles suivantes s'appliquent :

- éviter les événements répétitifs sans valeur opérationnelle ;
- privilégier les synthèses pour les traitements fréquents ;
- limiter le contexte aux données utiles ;
- surveiller les augmentations anormales de volume ;
- éviter qu'une écriture de journal ne dégrade excessivement un traitement métier ;
- prévoir une stratégie de purge ou d'archivage adaptée ;
- désactiver ou limiter les journaux de débogage en production.

Une optimisation ne doit jamais conduire à supprimer la traçabilité d'un événement critique.

---

# 16. Supervision et exploitation

Les erreurs et événements critiques doivent pouvoir être identifiés rapidement.

La plateforme doit permettre de repérer notamment :

- une répétition anormale d'erreurs ;
- un traitement durablement en échec ;
- un service externe indisponible ;
- une augmentation inhabituelle du volume d'événements ;
- une opération administrative sensible ;
- un événement de sécurité nécessitant une analyse ;
- l'absence inattendue d'événements attendus ;
- un échec du mécanisme de journalisation.

La V1.1 définit le cadre commun de journalisation et de consultation.

Les mécanismes avancés d'alerte, de supervision et d'analyse automatique pourront être enrichis ultérieurement sans modifier les principes du présent document.

# 17. Responsabilités

## 17.1 AKS Core

AKS Core est responsable :

- du modèle commun ;
- des catégories et niveaux partagés ;
- des règles de sécurité ;
- du mécanisme centralisé ;
- de la gestion de la corrélation ;
- de la conservation ;
- de la mise à disposition pour l'administration ;
- de la cohérence entre les modules ;
- des mécanismes de masquage communs ;
- du comportement en cas de défaillance de la journalisation.

## 17.2 Modules métier

Chaque module est responsable :

- d'identifier ses événements significatifs ;
- d'utiliser les catégories et niveaux communs ;
- de fournir un type d'événement stable ;
- de ne pas exposer de données interdites ;
- de fournir un contexte utile et proportionné ;
- de propager l'identifiant de corrélation ;
- de documenter ses événements obligatoires ;
- de maintenir la compatibilité avec AKS Core.

## 17.3 Administration

Les administrateurs autorisés sont responsables :

- de l'utilisation appropriée des journaux ;
- du respect de la confidentialité ;
- de l'analyse des événements importants ;
- de l'application des règles d'archivage et de suppression ;
- du signalement des anomalies détectées ;
- du contrôle des exports ;
- de la limitation des accès devenus inutiles.

## 17.4 Product Owner

Le Product Owner est responsable :

- de valider la politique générale de journalisation ;
- d'arbitrer les durées de conservation fonctionnelles ;
- d'approuver les exceptions significatives ;
- de s'assurer que les nouveaux modules respectent le présent cadre.

---

# 18. Gestion des changements et compatibilité

Toute évolution du modèle de journalisation doit préciser :

- le besoin ;
- les champs ajoutés, modifiés ou retirés ;
- l'impact sur les modules existants ;
- l'impact sur la consultation et les exports ;
- les règles de migration ;
- la compatibilité avec les événements déjà conservés ;
- les tests nécessaires ;
- les conditions de retour arrière.

Les champs et types d'événements existants doivent rester compatibles lorsque cela est raisonnablement possible.

Un type d'événement ne doit pas changer de signification sans versionnement ou migration explicite.

---

# 19. Gestion des exceptions

Toute exception aux règles de `LOG-001` doit être :

- motivée par un besoin identifié ;
- limitée à un périmètre précis ;
- évaluée au regard de `SECURITY-001` ;
- approuvée par le Product Owner ;
- documentée dans le document du module concerné ou dans une décision d'architecture ;
- assortie, lorsque nécessaire, d'une date de réexamen.

Une exception ne peut jamais autoriser la journalisation en clair d'un secret, d'un mot de passe, d'un jeton ou des réponses détaillées au questionnaire santé.

---

# 20. Éléments exclus

Le présent document ne définit pas :

- un système de supervision d'entreprise ;
- un centre opérationnel de sécurité ;
- une plateforme externe de centralisation de logs ;
- un moteur avancé d'analyse comportementale ;
- une conservation illimitée ;
- une copie complète des données métier dans les journaux ;
- la journalisation d'environnements personnels sans lien avec AKS Platform.

L'environnement Proxmox personnel du Product Owner est hors périmètre d'AKS Platform.

---

# 21. Critères d'acceptation

Le système de journalisation est conforme à `LOG-001` lorsque :

- tous les modules utilisent le mécanisme commun ;
- les événements importants sont identifiables ;
- les niveaux sont appliqués de manière cohérente ;
- les catégories permettent un filtrage fonctionnel ;
- les événements possèdent un type stable ;
- les traitements liés peuvent être rapprochés par corrélation ;
- les erreurs fournissent un contexte suffisant au diagnostic ;
- les secrets et données interdites ne sont jamais exposés ;
- le masquage est appliqué avant l'enregistrement ;
- les réponses détaillées du questionnaire santé ne sont jamais journalisées ;
- les données personnelles enregistrées sont limitées au strict nécessaire ;
- les journaux sont consultables depuis l'administration selon les droits ;
- les opérations sensibles sont traçables ;
- les opérations portant sur les journaux sont elles-mêmes tracées ;
- la conservation, l'archivage et la suppression sont maîtrisés ;
- les volumes restent compatibles avec l'environnement d'exécution ;
- les journaux de débogage sont limités en production ;
- un nouveau module peut s'intégrer sans créer son propre système isolé ;
- une évolution du modèle conserve une stratégie de compatibilité ;
- les exceptions sont explicitement documentées et approuvées.

---

# 22. Vérifications minimales

Avant validation d'un module ou d'une évolution transverse, les contrôles suivants doivent être réalisés :

1. vérifier que les événements obligatoires sont émis ;
2. vérifier que les niveaux sont adaptés à l'impact réel ;
3. vérifier que l'identifiant de corrélation est propagé ;
4. vérifier qu'aucun secret ou contenu interdit n'est enregistré ;
5. vérifier le masquage des données sensibles ;
6. vérifier les droits de consultation ;
7. vérifier le comportement lorsque la journalisation échoue ;
8. vérifier la stratégie de conservation et de purge ;
9. vérifier la compatibilité avec les modules existants ;
10. vérifier la lisibilité des événements depuis l'administration.

Elle assure la traçabilité, facilite le diagnostic, renforce la sécurité et améliore l'exploitation de la plateforme. Son efficacité repose sur un modèle centralisé, structuré, proportionné et respectueux de la confidentialité.
