# UX-001
# Principes d'expérience utilisateur d'AKS Platform

Version : 1.1  
Statut : Validé  
Version du produit : V1.1

---

# 1. Objet

Le présent document définit les principes d'expérience utilisateur applicables à l'ensemble d'AKS Platform.

Il constitue la référence commune pour concevoir, développer, valider et faire évoluer les interfaces publiques, administratives et métier de la plateforme.

L'objectif est de garantir une expérience cohérente, simple, accessible et fiable, quel que soit le module, le terminal utilisé ou le niveau de maîtrise de l'utilisateur.

---

# 2. Objectifs

L'expérience utilisateur doit permettre de :

- comprendre rapidement la finalité d'un écran ;
- identifier sans ambiguïté l'action principale attendue ;
- limiter le nombre d'étapes nécessaires ;
- réduire les risques d'erreur ou d'abandon ;
- conserver une cohérence entre les modules ;
- rester utilisable sur ordinateur, tablette et smartphone ;
- fournir un retour clair après chaque action ;
- rendre les fonctions d'administration exploitables sans connaissance technique avancée ;
- préparer l'ajout de nouveaux modules sans multiplier les comportements spécifiques.

---

# 3. Utilisateurs concernés

Les interfaces peuvent notamment être utilisées par :

- le public ;
- les licenciés ou leurs représentants légaux ;
- les professeurs ;
- les bénévoles autorisés ;
- les administrateurs de la plateforme ;
- les responsables du club.

Chaque interface doit être adaptée au contexte, aux droits et aux besoins réels de l'utilisateur concerné.

Une fonction non disponible pour un utilisateur ne doit pas être présentée comme accessible.

---

# 4. Principes directeurs

Les interfaces doivent respecter les principes suivants :

- simplicité ;
- cohérence ;
- lisibilité ;
- accessibilité ;
- prévisibilité ;
- sobriété ;
- prévention des erreurs ;
- retour immédiat après une action ;
- affichage des seules informations utiles au contexte ;
- continuité entre les différents modules.

La conception doit privilégier les usages réels plutôt que la multiplication de fonctions ou d'options.

---

# 5. Hiérarchie de l'information

Chaque écran doit présenter une hiérarchie claire.

Il doit être possible d'identifier rapidement :

- le titre de l'écran ;
- son objectif ;
- les informations principales ;
- l'action principale ;
- les actions secondaires ;
- l'état courant du traitement ou de l'objet consulté.

Les informations secondaires, techniques ou rarement utilisées doivent être regroupées ou présentées de manière progressive.

Une page ne doit pas afficher simultanément plus d'informations que nécessaire à la décision ou à l'action attendue.

---

# 6. Navigation

La navigation doit :

- conserver une organisation stable entre les modules ;
- indiquer clairement la position de l'utilisateur ;
- limiter la profondeur des menus ;
- rendre les fonctions principales directement accessibles ;
- distinguer les espaces publics, privés et administratifs ;
- éviter les impasses de navigation ;
- permettre un retour compréhensible vers l'écran précédent ou l'espace principal.

Les intitulés de menu doivent utiliser le vocabulaire fonctionnel du club et non des termes purement techniques.

Les actions importantes ne doivent pas dépendre uniquement d'une icône non accompagnée d'un libellé ou d'une aide explicite.

---

# 7. Parcours utilisateur

Un parcours doit être conçu comme une succession d'étapes compréhensibles et prévisibles.

Pour chaque parcours important, il convient de définir :

- le point d'entrée ;
- l'objectif utilisateur ;
- les informations nécessaires ;
- les étapes obligatoires ;
- les validations ;
- le résultat attendu ;
- les erreurs possibles ;
- le comportement en cas d'interruption ou de reprise.

Les parcours publics, notamment les formulaires, doivent limiter les abandons et éviter toute demande d'information non indispensable.

Les actions administratives fréquentes doivent pouvoir être réalisées sans navigation excessive.

---

# 8. Formulaires

Les formulaires doivent :

- limiter les champs au strict nécessaire ;
- identifier clairement les champs obligatoires ;
- utiliser des libellés explicites ;
- proposer le format le plus adapté à la donnée attendue ;
- valider les données au moment opportun ;
- afficher les erreurs à proximité du champ concerné ;
- préserver les informations déjà saisies lorsqu'une erreur survient ;
- éviter de demander plusieurs fois la même information ;
- confirmer clairement la prise en compte de l'envoi.

Les valeurs par défaut ne doivent être utilisées que lorsqu'elles sont sûres et compréhensibles.

Une action irréversible ou sensible doit faire l'objet d'une confirmation adaptée au risque.

---

# 9. Actions et boutons

L'action principale d'un écran doit être identifiable immédiatement.

Les conventions suivantes doivent être respectées :

- un libellé doit décrire l'action réalisée ;
- les verbes vagues tels que « OK » ou « Valider » doivent être évités lorsqu'un libellé plus précis est possible ;
- les actions secondaires doivent être visuellement distinctes ;
- les actions destructrices doivent être clairement identifiées ;
- un bouton indisponible doit expliquer, lorsque nécessaire, pourquoi l'action n'est pas possible ;
- une action en cours doit empêcher les doubles soumissions lorsqu'elles peuvent produire un doublon.

---

# 10. États d'interface

Toute interface dynamique doit prévoir les états nécessaires à sa compréhension.

Les principaux états sont :

- chargement ;
- aucun résultat ;
- succès ;
- avertissement ;
- erreur ;
- accès interdit ;
- indisponibilité temporaire ;
- traitement en cours ;
- données partielles ou obsolètes lorsque cette situation est possible.

Un écran vide ne doit pas être laissé sans explication.

Chaque état doit indiquer clairement ce qui se passe et, lorsque cela est possible, l'action suivante proposée à l'utilisateur.

---

# 11. Messages et retours utilisateur

Les messages doivent être :

- compréhensibles ;
- concis ;
- orientés vers l'action ;
- adaptés au contexte ;
- non techniques lorsque l'information technique n'est pas utile à l'utilisateur.

Un message d'erreur doit préciser :

- ce qui n'a pas fonctionné ;
- ce qui a été conservé ou non ;
- ce que l'utilisateur peut faire ;
- comment obtenir de l'aide lorsque le problème ne peut pas être corrigé directement.

Les identifiants techniques, traces, secrets ou détails internes ne doivent jamais être exposés au public.

Un message de succès ne doit être affiché que lorsque l'action a réellement abouti.

---

# 12. Prévention et gestion des erreurs

La plateforme doit prévenir les erreurs autant que possible par :

- des choix explicites ;
- des contraintes de saisie adaptées ;
- des contrôles avant enregistrement ;
- des confirmations proportionnées au risque ;
- la désactivation temporaire d'actions incompatibles ;
- la conservation des données déjà saisies ;
- l'affichage d'informations contextuelles.

Les erreurs ne doivent pas être imputées à l'utilisateur par une formulation culpabilisante.

Lorsqu'une opération peut être annulée ou corrigée simplement, cette possibilité doit être privilégiée à une confirmation systématique.

---

# 13. Accessibilité

Les interfaces doivent viser un niveau d'accessibilité compatible avec les usages et les moyens du projet.

Elles doivent notamment :

- offrir un contraste suffisant ;
- ne pas utiliser la couleur comme seul moyen de transmettre une information ;
- conserver des tailles de texte lisibles ;
- être utilisables au clavier lorsque le contexte le permet ;
- présenter un ordre de navigation logique ;
- associer les champs à leurs libellés ;
- fournir des textes alternatifs aux images porteuses d'information ;
- limiter les animations inutiles ;
- utiliser une terminologie simple et stable ;
- conserver des zones d'action suffisamment grandes sur mobile et tablette.

L'accessibilité doit être intégrée dès la conception et non ajoutée uniquement après développement.

---

# 14. Responsive et usages mobiles

Les écrans doivent rester exploitables sur :

- ordinateur ;
- tablette ;
- smartphone.

La consultation mobile est prioritaire pour les usages terrain et les formulaires publics.

L'adaptation responsive doit notamment garantir :

- l'absence de défilement horizontal non justifié ;
- la lisibilité des contenus ;
- l'accessibilité des actions principales ;
- la réorganisation cohérente des colonnes ;
- l'utilisation possible sans zoom obligatoire ;
- la conservation du contexte lors du passage d'une orientation à une autre.

Les tableaux volumineux doivent disposer d'une présentation adaptée aux petits écrans ou d'une solution alternative de consultation.

---

# 15. Cohérence visuelle et fonctionnelle

Les composants communs doivent conserver :

- les mêmes conventions de boutons ;
- les mêmes couleurs fonctionnelles ;
- les mêmes styles de messages ;
- les mêmes icônes pour une même fonction ;
- les mêmes comportements ;
- les mêmes règles de validation ;
- les mêmes conventions de date, heure et formatage.

La couleur principale du club peut être utilisée pour l'identité visuelle, sans compromettre la lisibilité ni la distinction des états fonctionnels.

Un module ne doit pas introduire un composant spécifique lorsqu'un composant commun répond déjà au besoin.

---

# 16. Données, listes et tableaux

Les listes et tableaux doivent permettre de comprendre rapidement les informations présentées.

Ils doivent, lorsque cela est pertinent :

- afficher les colonnes réellement utiles ;
- permettre le tri, le filtrage ou la recherche ;
- indiquer le nombre de résultats ;
- signaler clairement l'absence de données ;
- conserver les filtres pendant la navigation lorsque cela améliore l'usage ;
- distinguer les statuts par un texte explicite ;
- éviter d'exposer des données personnelles non nécessaires.

Les actions disponibles sur une ligne doivent être compréhensibles et ne pas provoquer d'opération sensible sans contrôle adapté.

---

# 17. Performances perçues

L'interface doit informer l'utilisateur lorsqu'un traitement nécessite un délai perceptible.

Les principes suivants s'appliquent :

- afficher un indicateur de chargement ;
- éviter les clics répétés ;
- ne pas bloquer l'ensemble de l'interface lorsqu'une zone seulement est concernée ;
- afficher progressivement les informations lorsque cela est pertinent ;
- confirmer la fin du traitement ;
- expliquer toute attente anormalement longue.

La performance perçue fait partie de l'expérience utilisateur, même lorsque le traitement technique est correct.

---

# 18. Sécurité et confidentialité dans l'interface

L'interface doit respecter les principes de sécurité et de confidentialité de la plateforme.

Elle doit notamment :

- masquer les secrets ;
- ne pas afficher de données sensibles sans nécessité ;
- respecter les droits d'accès ;
- éviter de révéler l'existence d'informations interdites à l'utilisateur ;
- ne pas conserver de données confidentielles dans des messages ou journaux visibles ;
- signaler clairement les expirations de session ou pertes d'autorisation.

Une interface ne constitue jamais une mesure de sécurité suffisante : les contrôles doivent également être appliqués côté traitement.

---

# 19. Validation UX

Toute nouvelle interface ou évolution significative doit être vérifiée au regard du présent document.

La validation doit au minimum contrôler :

- la compréhension de l'objectif de l'écran ;
- la visibilité de l'action principale ;
- la cohérence avec les autres modules ;
- le comportement sur mobile ;
- les états de chargement, succès, erreur et absence de résultat ;
- la prévention des doubles actions ;
- la clarté des messages ;
- l'accessibilité des éléments essentiels ;
- la protection des informations sensibles.

Lorsque cela est pertinent, une validation par un utilisateur représentatif doit compléter la revue fonctionnelle.

---

# 20. Éléments exclus

Les éléments suivants sont exclus du périmètre de `UX-001` :

- la définition graphique détaillée de chaque écran ;
- la création d'une bibliothèque de composants complète sans besoin identifié ;
- les règles métier propres à chaque module ;
- les procédures techniques de développement ;
- les choix imposés par une infrastructure personnelle extérieure à AKS Platform.

Les documents spécifiques aux modules peuvent compléter ces principes sans les contredire.

---

# 21. Critères d'acceptation

Une interface est considérée comme conforme lorsque :

- son objectif est identifiable immédiatement ;
- l'action principale est claire ;
- la navigation est cohérente avec le reste de la plateforme ;
- les formulaires limitent les erreurs et préservent les données saisies ;
- les états de chargement, succès, erreur et absence de résultat sont traités ;
- les messages sont compréhensibles et orientés vers une solution ;
- l'interface reste exploitable sur ordinateur, tablette et smartphone ;
- les informations sensibles ne sont pas exposées ;
- les composants communs conservent les mêmes comportements ;
- les principales actions sont accessibles sans connaissance technique avancée ;
- aucune évolution UX ne casse un parcours validé de la V1.0.0.

# 23. Conclusion

Les principes définis dans `UX-001` garantissent une expérience homogène, accessible et durable pour l'ensemble des utilisateurs d'AKS Platform.

Toute nouvelle interface doit s'appuyer sur ces règles afin de préserver la cohérence globale de la plateforme, de réduire les erreurs et de faciliter l'arrivée des futurs modules métier.
