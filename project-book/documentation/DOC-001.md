# DOC-001
# Règles de documentation d'AKS Platform

Version : 1.2.0
Statut : Validé
Dernière mise à jour : 2026-07-23
Version du produit : V1.1

---

# 1. Objet

Ce document définit les règles générales de production, d'organisation, de maintenance et d'évolution de la documentation officielle d'AKS Platform.

Le Project Book constitue la référence documentaire du produit.

Il est lui-même considéré comme un système documentaire : ses documents sont organisés selon des responsabilités explicites, des dépendances maîtrisées et des interfaces documentaires constituées par les références croisées.

---

# 2. Objectifs

La documentation doit :

- être la source officielle des décisions fonctionnelles ;
- rester synchronisée avec le produit ;
- éviter les duplications ;
- faciliter la maintenance ;
- permettre l'arrivée de nouveaux contributeurs ;
- grandir sans devenir plus complexe à comprendre ou à utiliser.

Toute évolution du Project Book doit améliorer sa lisibilité globale au moins autant qu'elle améliore son contenu.

---

# 3. Organisation

Le Project Book est organisé par domaines :

- roadmap ;
- architecture ;
- administration ;
- documentation ;
- modules ;
- release ;
- ux.

Chaque document possède une responsabilité unique.

Le système documentaire repose sur quatre couches :

| Couche | Question principale |
|---|---|
| Produit | Pourquoi construit-on cette plateforme ? |
| Gouvernance | Comment décide-t-on de son évolution ? |
| Documentation | Comment capitalise-t-on et retrouve-t-on la connaissance ? |
| Implémentation | Comment construit-on techniquement la plateforme ? |

Chaque couche répond à une famille de questions distincte. Chaque document répond à une seule question appartenant à la famille correspondant à son périmètre.

---

# 4. Hiérarchie documentaire

La hiérarchie des documents structurants est la suivante :

```text
VISION-001
    ↓
OBJECTIVES-001 et SCOPE-001
    ↓
ROADMAP-001 et GOV-001
    ↓
ARCH-001
    ↓
CORE-001
    ↓
Documents transverses, administration et UX
    ↓
Modules métier
    ↓
Guides opérationnels et documents de release
```

`INDEX-001` constitue le catalogue documentaire officiel et fait autorité sur l'organisation du corpus.

En cas de divergence, le document spécialisé fait autorité sur son propre périmètre, sous réserve de rester conforme aux documents directeurs situés au-dessus de lui dans cette hiérarchie.

Les standards spécialisés appliquent les principes généraux définis dans le présent document à un périmètre déterminé. Ils ne les remplacent pas et ne doivent pas dupliquer leur contenu.

---

# 5. Convention de nommage

Les documents utilisent le format :

```text
TYPE-NNN
```

Exemples :

- ROADMAP-001
- ARCH-001
- CORE-001
- ADMIN-001
- CONFIG-001
- LOG-001
- UX-001
- RELEASE-001
- STD-001

---

# 6. Principe de responsabilité documentaire

Chaque document du Project Book doit répondre à une seule question clairement identifiable.

Cette question ne doit recevoir sa réponse complète dans aucun autre document officiel.

Un document ne peut être obligatoire que si sa question et sa responsabilité sont clairement distinctes de celles des autres documents obligatoires.

Un document standard peut définir qu'un autre document existe, préciser son rôle ou le référencer. Il ne doit jamais décrire à sa place le contenu détaillé qui relève de la responsabilité de cet autre document.

La création d'un document est justifiée par l'apparition d'une responsabilité documentaire nouvelle, jamais par la volonté d'augmenter artificiellement le niveau de détail ou le nombre de documents.

---

# 7. Création d'un nouveau document

Avant de créer un document, les trois questions suivantes doivent être examinées :

1. Quelle question nouvelle doit-il traiter ?
2. Cette question est-elle déjà couverte ailleurs ?
3. La réponse mérite-t-elle un document autonome ?

Un nouveau document ne doit être créé que lorsque :

- la question est nouvelle ;
- elle n'est pas déjà couverte par un document existant ;
- son périmètre justifie une responsabilité autonome.

Lorsqu'une information complète une responsabilité existante, le document concerné doit être enrichi plutôt qu'un nouveau document créé.

---

# 8. Contenu minimal

Chaque document doit comporter, lorsque ces éléments sont applicables :

- un objet ;
- son périmètre ;
- ses dépendances ;
- ses exclusions si nécessaire ;
- ses critères d'acceptation ;
- ses références.

Le contenu doit rester limité à la question et à la responsabilité propres au document.

---

# 9. Cycle de vie

Toute évolution documentaire suit le processus suivant :

1. analyse ;
2. rédaction ;
3. revue ;
4. validation ;
5. intégration Git ;
6. publication.

Aucun développement majeur ne doit être réalisé sans documentation lorsque celle-ci est requise.

Toute règle de gouvernance documentaire doit réduire durablement la complexité du Project Book ou, au minimum, ne pas augmenter sa complexité cognitive pour les futurs contributeurs.

---

# 10. Règles de mise à jour

Les mises à jour doivent :

- conserver l'historique fonctionnel ;
- éviter les informations contradictoires ;
- modifier uniquement les documents concernés ;
- préserver les références croisées ;
- maintenir une faible duplication et une forte cohésion documentaire ;
- ne pas introduire un niveau d'abstraction sans bénéfice concret pour la lisibilité ou la maintenance.

Tout document officiel créé, renommé, déplacé, validé, rendu obsolète, archivé ou remplacé doit entraîner la mise à jour de `INDEX-001` dans la même livraison documentaire.

Les anciens documents PB-* restent disponibles pendant la migration puis pourront être supprimés une fois leur contenu repris.

---

# 11. Qualité

Chaque document doit être :

- clair ;
- concis ;
- cohérent ;
- fortement cohésif ;
- faiblement couplé aux autres documents ;
- sans duplication inutile ;
- indépendant des détails d'implémentation lorsque ceux-ci ne relèvent pas de sa responsabilité.

Les références croisées constituent les interfaces documentaires du Project Book. Elles doivent être explicites, nécessaires et cohérentes.

---

# 12. Contrôle qualité avant validation

Avant validation, chaque document doit faire l'objet des contrôles suivants :

- cohérence fonctionnelle avec la vision, les objectifs, le périmètre et la roadmap ;
- cohérence des références croisées et existence des documents cités ;
- cohérence du numéro de version, du statut et de la date de mise à jour ;
- cohérence avec les règles du Project Book et avec `INDEX-001` ;
- absence de contradiction, de duplication inutile ou de contenu tronqué ;
- identification explicite des dépendances, exclusions et critères d'acceptation applicables ;
- identification claire de la question à laquelle le document répond ;
- confirmation que cette question n'est pas déjà traitée ailleurs ;
- confirmation que le document améliore ou préserve la lisibilité globale du corpus.

Un document ne peut être déclaré `Validé` tant que ces contrôles ne sont pas satisfaits.

---

# 13. Critères d'acceptation

La documentation est conforme lorsque :

- chaque sujet possède un document de référence ;
- chaque document répond à une question unique ;
- une même question ne reçoit pas plusieurs réponses concurrentes ;
- les références sont cohérentes ;
- les dépendances sont identifiées ;
- les documents sont versionnés ;
- les décisions validées sont documentées ;
- les évolutions n'augmentent pas inutilement la complexité cognitive ;
- `INDEX-001` reflète fidèlement le corpus documentaire actif.

# 16. Conclusion

Le Project Book constitue la mémoire fonctionnelle d'AKS Platform.

Il doit évoluer comme un système cohérent : chaque document possède une responsabilité unique, chaque référence constitue une interface explicite et toute nouvelle règle doit améliorer durablement la lisibilité, la maintenabilité et l'évolutivité du corpus.

La qualité du Project Book ne se mesure pas uniquement à la richesse de son contenu, mais également à sa capacité à grandir sans devenir plus difficile à comprendre ou à utiliser.
